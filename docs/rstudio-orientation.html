RStudio Orientation

Before installing the package, it’s worth a few minutes getting comfortable with RStudio itself — where things are, and how your data looks once it’s loaded. If you already work in RStudio day to day, you can skim this and move on to Install jstats.

Start a Project

Before installing the package or loading any data, create an RStudio Project.

Without a Project, R needs to be told where your files live (its “working directory”), and that quickly drags in folder paths, the difference between \ and /, and the fact that Windows and Mac write paths differently. A Project sidesteps all of it: it ties a folder, your data, and your work together, so R always knows where it is.

To create one, in RStudio choose File → New Project → New Directory → New Project, give it a name (for example jstats-practice), pick where it should live, and click Create Project. RStudio opens your new Project.

From now on, you reopen this Project by double-clicking its .Rproj file (the small file RStudio created inside your Project folder). That one habit — always work inside a Project — saves more beginner headaches than any other.

The RStudio panes

With your Project open, you’ll see three panes to begin with. A fourth — the source pane, in the top-left — appears the first time you open a script or view a dataset. Here’s what each is for:

  • Editor / source (top-left) — where you write and save your code, and where a dataset opens as a grid when you click it to view. This is your reusable list of commands.
  • Console (bottom-left) — where code runs and results print.
  • Environment (top-right) — lists the data and objects currently loaded, by name and size (not the data itself). Importantly, what lives here is a working copy of your data, held in memory. Mistakes hit the copy, never the original file on disk.
  • Files / Plots / Help (bottom-right) — your folder contents, any charts you make, and help pages.

Let’s make the panes concrete with a dataset that already comes with R. R ships with several built-in datasets for practice and examples; one is mtcars (data on some classic car models). You can copy it into a name of your own choosing using R’s assignment operator, <-, which means “put what’s on the right into the name on the left”:

cars <- mtcars

You now have a dataset called cars — the same data, under a name you picked. Look at the Environment pane (top-right): cars appears, described as 32 obs. of 11 variables. That’s its name and size — the Environment lists what you’ve loaded, not the data themselves. To see the data, click cars in the Environment. A tab opens in the source pane (top-left) showing a grid — rows of cars down the side, variables across the top. This is the same top-left pane where your scripts will open shortly; RStudio uses it for both.

iris is another built-in dataset (measurements of flowers). Because it’s built in, it’s already available by its own name — you don’t have to copy it first. To take a quick look, run:

View(iris)

Note the capital V — View is one of a handful of base-R functions that breaks the usual all-lowercase habit (a leftover from R’s long, many-handed history rather than any real reason), so view(iris) won’t work; R distinguishes capitals from lowercase. jstats keeps its own functions lowercase, in step with the base-R majority.

The same kind of grid opens. Notice one column, Species, holds group names (setosa, versicolor, virginica) instead of numbers — your first glimpse of a categorical variable: a column that sorts cases into groups. You’ll meet these properly later.

You’ll see mtcars and iris again and again in R’s help pages and in other packages’ examples — now you know what they are, so they won’t catch you off guard. jstats brings its own example dataset too, community, which becomes available by name once you install and load the package — we’ll use it from the Quick Start onward.

TipComing from SPSS or Stata?

A rough map if you’ve mostly used SPSS through its menus: the Console is like the output/viewer window where results appear, and the Plots area is where charts show up, much as they do in SPSS output. The closest thing to Data View is the read-only grid that opens when you click a dataset in the Environment — but note the Environment pane itself is not Data View: it only lists what’s loaded, by name and size. The Editor is the one piece without a close menu-equivalent — it’s your saved list of typed commands (nearest to the SPSS syntax window, which menu users rarely open).

You’ll also notice Terminal and Background Jobs tabs near the Console. These are for more advanced use — the Terminal is a direct line to your computer’s command prompt (PowerShell on Windows, the old DOS-style window) — and you can ignore them for now.

One small thing that trips up newcomers: R is case-sensitive. Income and income are two different names to R. When something doesn’t work, a stray capital is a common culprit.

Try the console

Let’s make the Console feel real, since it’s where everything ultimately runs. Click in the Console (bottom-left), type a sum, and press Enter:

2 + 2

R answers straight back:

[1] 4

The console is a calculator that talks back. Now try a function — a named command that takes some input in parentheses and hands you a result. This one finds a square root:

sqrt(144)
[1] 12

That’s the shape of nearly everything you’ll do in R: a function name, inputs in parentheses, a result back. (And yes, sqrt is plain R — you’re already learning the real thing.)

So what’s the script (the Editor) for, if the console can do everything? Just this: the console is for doing, the script is for keeping. Anything you type in the console runs and then scrolls away — perfect for quick, one-off jobs like a fast calculation or installing a package. When you’re doing analysis you’ll want to keep — and re-run, and share — you type those commands in the script and save the file. We’ll start doing exactly that once there’s something worth saving.

Start each session with a clean slate

One quick setting is worth changing now. By default, when you quit RStudio it offers to save everything currently loaded — your whole Environment — and reload it next time. That sounds convenient, but it causes more confusion than it saves: old objects from a previous session linger, and you can’t easily tell what you meant to keep versus what’s just left over. It’s a classic source of “why is this still here?” puzzlement.

The cleaner habit is to start every session empty and rebuild what you need by re-running your script — which is the dependable record of your work anyway. To set that up, go to Tools → Global Options → General, then:

  • uncheck Restore .RData into workspace at startup, and
  • set Save workspace to .RData on exit to Never.

Now each time you open RStudio you start fresh, and your script — not whatever happened to be in memory last time — is the source of truth.
