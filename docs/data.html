Loading, Saving, Converting Data

This page is a hands-on run-through of moving data in and out of jstats: loading files, saving your work back out, and converting between formats.

NoteBefore you start

To follow along you’ll need:

  • R, RStudio, and jstats installed. If they’re not, work through Get Started, Steps 1-5 first and come back.
  • An RStudio Project open, so your working directory and files are already set up.
  • A new, empty R script open (File > New File > R Script). You’ll copy each numbered block into it, one at a time, running each before moving on.
NoteHow to Use this Page

The blocks on this page build one script. Paste each block into the same R script file as you reach it, in order — by the end, the whole file runs cleanly from top to bottom, except in the few places where a comment says an error is deliberate (so you can see what one looks like).

A few mechanics worth knowing:

  • Every block starts with a dashed comment header (## ---- 1. ... ----). It marks where one block ends and the next begins in your growing script — and RStudio reads those headers as sections, so they appear in the script’s outline for easy jumping around.
  • Experiment freely as you go — change a number, rerun a line, poke at things. Building the script yourself is the point.
  • Prefer to grab everything at once? The complete, clean script – without your experiments, and finished so it runs top to bottom – is available to download at the bottom of the page.

This box is the same on every hands-on page in this group, so once you’ve read it here, you can skip it elsewhere.

TipNew to R?

You don’t need to memorize any of this yet. Right now the goal is simply to run each line and watch what happens – the ideas settle in as you go. A few things R doesn’t care about, so you don’t have to either: which line numbers your code lands on, and how many blank lines sit between commands – paste each block anywhere below your previous code and it all works the same. Lines starting with # are comments: notes for humans that R skips entirely (extra hash marks, like ##, are still just a comment). And running a line a second time is almost always harmless – you’ll simply get the same result again. If a line doesn’t work, check that jstats loaded without errors, then try it again.

1. Load the package and a dataset

We’ll start by loading the package and one of its built-in practice datasets.

## ---- 1. Load the package and a dataset ---------------------------------

## Load the jstats library
library(jstats)

## Load the package dataset called community
## It's loaded as a data frame into the Environment
jload("community")
NoteWhat’s the Base R equivalent?

For a dataset that ships inside a package, base R has one too: data(community) loads it. The difference shows up beyond package data: data() only reaches datasets bundled in packages, while jload() is one loader for everything on this page — package datasets, your saved .rds files, and SPSS, Stata, SAS, Excel, and CSV files — with the same one-line call.

2. Objects and the assignment operator

Anything you create in R is stored as an object. Here we pull a single variable out of the dataset and give it a name of our own.

## ---- 2. Objects and the assignment operator ----------------------------

## "Objects" other than data frames can go into the Environment
## The "Assignment Operator" (less-than followed by a dash) is a key Base R fundamental
## Take the thing on the right and assign it to what's on the left
## Take the Region variable from the community data frame and assign it to a new object
## The new object we'll call "MyObject"
MyObject <- community$Region ## (Base R code)

3. More than one dataset – and cleaning up

You’re not limited to one dataset at a time, and you can remove objects you no longer need.

## ---- 3. More than one dataset, and cleaning up -------------------------

## You can have more than 1 data frame in the Environment
jload("clinic")

## Remove MyObject from the Environment
## Note that this is a Base R function (j is not the first letter)
rm(MyObject)

## Remove community from the Environment
rm(community)

## Remove clinic from the Environment
rm(clinic)
TipComing from SPSS, Stata, or SAS?

SPSS can have several datasets open too – but each opens in its own Data Editor window, and switching the active one takes a syntax chain most users never bother with:

GET FILE='community.sav'.
DATASET NAME community.
GET FILE='clinic.sav'.
DATASET NAME clinic.
DATASET ACTIVATE community.

And when you’re finished with a dataset in SPSS, you close its Data Editor window or run DATASET CLOSE clinic. – the R counterpart is the rm() cleanup you see above.

In R there are no windows to juggle: the Environment holds several data frames side by side, and every command simply names the one it uses – the $ in community$Region means “the Region column, in the community dataset.”

4. How much output you see

jstats can show more or less detail. The output level controls how much appears.

## ---- 4. How much output you see ----------------------------------------

## The output level is set to "standard" by default
## After you learn more about jstats (and R) you might not want all of that output
joutput("minimal")

## With minimal output, you get less information
jload("community")

## Set the output level back to standard
joutput("standard")
NoteWhat’s the Base R equivalent?

There isn’t one. Base R has no output-level setting — each function prints whatever it prints, and how much you see varies with each package author’s choices. joutput() is a single, jstats-wide control.

5. Functions with no arguments, and getting help

Most functions need something inside the parentheses, but a few work on their own. Every function also has a help page.

## ---- 5. Functions with no arguments, and getting help ------------------

## A few functions work correctly without input between the parentheses, but most require input
jload()  # This will cause an error with a message about how you might fix the error

## But this one will show you the current output settings for jstats
joutput()

## Here's a Base R function that works without arguments/input
## List the objects/data frames in the Environment
ls()

## Here's another one from Base R
## A working directory is the "path" to where your project is stored on your computer
## If you're not working in a project, you have to set this yourself (finicky)
getwd()

## The technical name for a function's inputs are "Arguments"
## View help files describing necessary and optional Arguments by running...
?jload

## The same ? help works for base R functions and functions from any installed package
?ls

## And this one opens the package overview: every jstats function, grouped by purpose
?jstats
NoteWhat’s the Base R equivalent?

? is base R – ?jload, ?ls, and ?mean all work the same way, for base functions and for functions from any installed package. (In RStudio, putting the cursor on a function name and pressing F1 does the same thing. On Macs the key varies from machine to machine – F1 on some, fn + F1 on others, and on some Macs the function keys must first be enabled in System Settings.) What a base R setup doesn’t have is a grouped index: a typical workflow spans functions scattered across many packages, each with its own alphabetical help listing. ?jstats shows the whole toolkit on one page, organized by purpose – one advantage of the functions traveling together as a package.

6. The overwrite prompt

When you load a dataset that’s already open, jstats checks with you before overwriting it – watch for the y/n prompt in the Console window.

## ---- 6. The overwrite prompt -------------------------------------------

## The output you get in the Console Pane can include a confirmation prompt
## Here, jstats will not overwrite community unless you give it permission
jload("community") # Look at the y/n prompt in the console

## Set an optional overwrite argument/input to overwrite without the prompt
jload("community", overwrite = TRUE)
NoteWhat’s the point?

The overwrite argument is here to show you two things:

  • What an argument looks like. Options go inside the parentheses as name = value – here, overwrite = TRUE.
  • Prompts can be overridden. The few times jstats asks for confirmation, the prompt appears in the Console – and each one can be switched off with the right argument. That matters when code runs automatically, with nobody at the keyboard to answer: it happens in several places behind this website, and in the finished script you can download at the bottom of this page.
NoteWhat’s the Base R equivalent?

Base R doesn’t ask. Loading or assigning over an existing object replaces it silently — no prompt, no warning, and the old copy is gone. The confirmation prompt is a jstats safeguard.

7. Saving to a Data folder

Let’s save your work – first with no folder specified, to see where files go by default, and then to a Data folder of your own.

## ---- 7. Saving to a Data folder ----------------------------------------

## Use the assignment operator to copy community to a new data frame called MyData
MyData <- community

## Save it with no folder specified -- where does the file go?
## The console message shows the full path: it landed in your project's
## working directory, the default location
## It's the same path the Base R getwd() command showed you earlier
jsave(MyData, "WhereDidIGo.rds")

## Delete that test file
## file.remove() is base R -- the file-on-disk equivalent of the rm()
## you used for data frames. It replies with a cryptic [1] TRUE
file.remove("WhereDidIGo.rds")

## Now name a Data folder of your own
## jstats will create this folder upon first save
## Or you can create it manually
joptions(data.dir = "Data")

## Save the dataset to the Data folder and call it MyData_r.rds
## Specify R native format with the .rds extension
## The Data folder is created now, on this first save
jsave(MyData, "MyData_r.rds")

## Remove MyData and community data frames from the environment
## Does not remove the saved file from the Data folder
rm(MyData)
rm(community)
NoteWhat’s the point?

Why set a data folder at all? Tidiness now, and a payoff later. With data.dir set, every save and load on this page knows where to look. And that’s the habit to keep: put joptions(data.dir = "Data") near the top of your own scripts, then drop a dataset of yours into that folder and jload("YourFile.sav") opens it by name alone. If you’re not sure where the folder is on your computer, the jsave message has already told you – it prints the full path each time, ready to paste into File Explorer (Windows) or find in Finder (Mac).

NoteWhat’s the Base R equivalent?

Saving to the working directory by default is base R’s behavior too – saveRDS(MyData, "MyData_r.rds") puts the file in the same place; jstats just tells you, in full, where it went. To use a folder, base R won’t create it for you (that’s dir.create("Data") first), and the folder name gets typed into every path from then on – with any subfolders along for the ride, e.g. saveRDS(MyData, "Data/CommunityStudy/Wave1/MyData_r.rds"). joptions(data.dir = ...) sets the location once, and every jsave() and jload() after that already knows where to look.

And note what file.remove() didn’t do: ask. Like the silent overwrite in block 6, base R deletes with no confirmation – mix up a filename and a real data file is gone. jstats’s prompts exist precisely because of edges like that.

8. Saving across formats – and missing values

Moving between file formats is easy, with one wrinkle around missing values that jstats handles for you.

## ---- 8. Saving across formats, and missing values ----------------------

## Reload the saved MyData.rds dataset from the data folder.
jload("MyData_r.rds")

## The original community dataset carries SPSS-style missing values
## So it saves to SPSS format (.sav) without a problem
jsave(MyData_r, "MyData_spss.sav")

## But the SPSS-style missing-value codes need to be converted
## if you want to save to a format other than R or SPSS
jsave(MyData_r, "MyData_stata.dta")  # This will draw an error

## Convert to Stata format (.dta)
MyData_stata <- jconvert(MyData_r, to = "stata")
jsave(MyData_stata, "MyData_stata.dta")  # Now no error
TipComing from SPSS, Stata, or SAS?

In SPSS, user-missing values are declared in Variable View and travel with the file. When you Save As a different format, those definitions don’t always carry across cleanly. jstats keeps the equivalent information with your data, and when a format needs it in a different form, jconvert() does the translation – which is why saving straight to Stata format above stops and asks you to convert first.

NoteWhat’s the Base R equivalent?

Base R alone can’t write SPSS or Stata files. You’d install and load the haven package and call haven::write_sav() or haven::write_dta() yourself — and haven is exactly the proven library jstats uses under the hood, so nothing here reinvents the wheel. What jstats spares you is the setup and the sharp edge: loading an extra package in every script, learning its separate function set, and handling the missing-value translation by hand — haven refuses SPSS-style missing codes headed into Stata format, just as you saw above, and converting them yourself is genuinely fiddly. jconvert() is that translation, done for you.

Two conversion habits are worth knowing even if you’re an experienced R user. Data that arrive with SPSS-style missing values are ordinary numbers underneath (a -99 really is a -99), so a base R function – or any package that doesn’t read the declarations the way jstats does – can mistake them for real values. Converting to Stata-style (jconvert(to = "stata")) stores them as a special kind of NA instead, which every R function already treats as missing. And for maximum compatibility, jconvert(to = "baseR") turns all declared missing codes – SPSS-, Stata-, or SAS-style – into plain NA. Even if you never use another jstats function, painless conversion across these formats is worth having.

9. Reloading, and how missing codes travel

Load the saved files back and see how each format keeps its missing-value codes.

## ---- 9. Reloading, and how missing codes travel ------------------------

## Remove all data frames from the environment
rm(list = ls())

## Load all data sets back
jload("MyData_r.rds")
jload("MyData_spss.sav")
jload("MyData_stata.dta")

## Examine how the missing codes are kept in the different formats
jfreq(MyData_r, Education)  # will print whatever was originally stored
jfreq(MyData_spss, Education)  # SPSS-style user-defined missing codes (UDMs)
jfreq(MyData_stata, Education) # Stata-style UDMs

MyData_r_with_stata_UDM <- jconvert(MyData_r, to = "stata")
jfreq(MyData_r_with_stata_UDM, Education) # Now the UDMs display in Stata format

10. Turning up the digits

One more control before the final check: how many decimal places jstats shows. The default setting is 3, the maximum is 7 – and for what comes next, we want all 7. (One exception: p-values keep their own fixed three-decimal convention and ignore this setting.)

## ---- 10. Turning up the digits -----------------------------------------

## By default, jstats shows up to 3 decimal places
jdesc(MyData_r, Age)   # the Age mean reads 40.65 -- the box below explains why not 40.650

## Turn the display up to the maximum of 7 decimal places
joutput(digits = 7)
jdesc(MyData_r, Age)   # the same mean now reads 40.6504854
TipComing from SPSS, Stata, or SAS?

SPSS output shows a fixed number of decimals, padding with zeros to fill the width – a mean of 40.65 appears as 40.650 in a three-decimal table. jstats goes the other way: it never prints trailing zeros, so digits sets a maximum, and each column shows only the decimals it actually needs. That’s why the mean above reads 40.65 at the default setting of 3 – its third decimal happens to be a zero. And if a statistic lands exactly on a round value, it displays short even at digits = 7: that’s trimming, not truncation or a stuck setting.

NoteWhat’s the Base R equivalent?

The closest is options(digits = ...), and it’s a different animal: it sets significant digits, globally, for nearly everything R prints. joutput(digits = ...) means decimal places, and it changes only jstats output.

11. The proof that nothing was lost

Those three data frames each made a round trip through a different file format. If the trips were truly lossless, the same statistics should come back from every one of them – right down to the seventh decimal.

## ---- 11. The proof that nothing was lost -------------------------------

## Ask for the mean of Age from each of the three reloaded data frames
jdesc(MyData_r, Age)
jdesc(MyData_spss, Age)
jdesc(MyData_stata, Age)
## All three report the identical mean: 40.6504854
## Same data, three file formats, nothing lost along the way

## And one base R caution, shown live (the box below explains):
## Income carries declared missing-value codes (-99, -98)
jdesc(MyData_spss, Income)   # the correct mean -- jdesc leaves the codes out
## Base R gives a bare, unlabeled number, so extra work is needed just to
## make it visible: cat() adds a label, and the "\n"s add blank lines to
## set it apart. Run line by line (interactively), a plain mean() prints
## only a cryptic [1] number; run with Source, it wouldn't appear at all
cat("\n\nBase R mean of Income:", mean(MyData_spss$Income), "\n\n\n")  # a poisoned mean

## Set the display back to the default before you go
joutput(digits = 3)

## Put the data folder setting back to its default (the working directory) too
## Passing "" clears it (data.dir = NULL would leave it unchanged)
## A finished script leaves your session exactly as it found it
joptions(data.dir = "")
TipComing from SPSS, Stata, or SAS?

jdesc() is the counterpart of SPSS’s Analyze > Descriptive Statistics > Descriptives – the same N, minimum, maximum, mean, and standard deviation you’d read from that output table. And the check above may feel familiar from practice: reopening a dataset that’s been through another format, and confirming the descriptives still match.

The matching numbers are only half the story – the missing-value declarations made the round trip too, which is arguably the bigger deal. You saw it back in block 9: Education’s codes come through meaning the same thing in every file, with the Stata copy simply wearing its .a/.b form. In SPSS terms, your MISSING VALUES definitions survived the Save As. (Age itself carries no missing codes – that’s exactly why it makes a clean test variable here.)

NoteWhat’s the Base R equivalent?

For a clean variable like Age, base R agrees exactly: mean(MyData_r$Age) returns the same 40.65049. The difference appears with variables that carry declared missing-value codes, like Income – those codes are ordinary numbers to base R, and the block’s last lines show the damage live: mean(MyData_spss$Income) folds the -99s and -98s straight into the average, while jdesc() reads the declarations and leaves the codes out automatically — provided the declarations exist: here they traveled with the files, and when a dataset arrives with codes that were never declared, you add the declarations yourself with jstats’s jdeclare_udm() function. It’s the same protection you’d expect from commercial statistical software.

NoteWhat’s the point?

The download at the bottom of this page is the finished version of what you built – and finished means something specific. A goal every experienced programmer and statistician takes for granted, but new researchers are rarely told: a completed script should run from beginning to end without errors. On this page you experimented – deliberate errors and prompts included, because that’s how working scripts develop. The download shows what the end of that process looks like, and it differs from the page blocks in three principled ways: the deliberate error lines are commented out, the y/n prompt line is commented out (a prompt would pause an unattended run), and the saves – plus the one reload that could collide with an earlier run’s leftovers – carry overwrite = TRUE so the whole file can be run again and again, exactly the automatic-execution override from block 6.

To prove a script is finished, use the Source icon (to the right of Run in the script toolbar): it runs the entire file, top to bottom, in one go. A few things to know. Source shows the output without echoing each command back to the Console – that’s normal – and the small arrow beside Source offers “Source with Echo” if you’d like the commands shown too. jstats functions print their own tables, so their output always appears; but a bare base R line that merely displays a value – ls(), getwd(), mean() – stays quiet under Source, because R only auto-prints results when you run lines interactively – and even interactively, what it prints is a bare, unlabeled number. That’s why the script prints its base R mean with cat(), a label, and a few blank lines to set it apart – hand-formatting that jstats tables do for you. And if you re-run the script you pasted together yourself, the saves will refuse to overwrite their earlier files – that’s block 6’s lesson again; add overwrite = TRUE just as the finished file does.

Nothing needs deleting between runs, and nothing is deleted: the overwrite arguments handle the repeats, and the Data folder with its three saved files stays put after every run – that’s what saving is for. Remove the files by hand if you ever want a clean slate.

One more setup idea if you’re keeping the file – and it’s worth keeping: finished scripts travel. Sooner or later you’ll share one with a supervisor or collaborator, or receive one from someone else, and a script that runs top to bottom on another person’s computer is the standard to aim for. So a downloaded script deserves a proper home: jstats created the Data folder for you during the run-through, but you can also make folders yourself in RStudio – in the Files tab of the lower-right pane, click New Folder and type a name into the popup. “Scripts” is a good one for storing this download, and it will appear right beside the Data folder you created by following the steps above.
